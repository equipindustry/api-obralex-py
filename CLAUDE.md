## Project

api-obralex-py is a FastAPI REST API for **Equip Construye**, a B2B marketplace for construction materials in Lima, Peru. It serves as the backend for MCP tools consumed by equip-mcp-hub and api-maia (the agentic orchestrator).

## Commands

```bash
# Run locally
uvicorn main:app --reload --port 8000

# Format
black .

# Docker build (for Cloud Run deployment)
docker build -t api-obralex --platform linux/amd64 .
```

No test framework is configured yet.

## Architecture

```
Client (Telegram) → api-maia → equip-mcp-hub → api-obralex → Vertex AI Search + Cloud Storage
```

### Key layers

- **`src/api/`** — FastAPI routers. Each file creates a `router` registered in `main.py` with prefix `/api/v1`.
- **`src/services/`** — Business logic. Services are instantiated at module level in the router files (not dependency-injected).
- **`src/models/`** — Pydantic response models.
- **`src/core/`** — Config (env vars via `Config` class), GCP credential setup (`environment.py`), logging.

### Services

- **`VertexAISearchService`** — Searches inventories via Google Discovery Engine. Returns `InventorySearchResult` dataclass with ~37 fields. Uses `MessageToDict` from protobuf to parse `struct_data`. Fields map 1:1 with the BigQuery table `inventories-simplify-prod` (singular names: `category`, `subcategory`).
- **`InventorySchemaService`** — Resolves which fields a customer must specify for a given product query. Combines Vertex AI Search (to identify category/subcategory) with a JSON schema cached from Cloud Storage (TTL-based). Resolution hierarchy: subcategory → category → default. If the JSON only contains `subcategory_schemas`, it auto-builds `category_schemas` by grouping subcategories by their `category` field (union of required_fields, merged options). Also exposes a full catalog of categories/subcategories with field details via `get_catalog()`.

### Terminology

- **Inventory** = material in startup's warehouse (what we have in stock). Source of truth: MongoDB `inventories` collection, synced to BigQuery (`inventories-simplify-prod`) via `api-adatrack` and indexed in Vertex AI Search.
- **Product** = material the customer requests. Each product links to 1 inventory.
- Endpoints and code use `inventories` naming, not `products`.
- Current dataset: 251 items focused on Acero and Electricidad categories. Expanding incrementally as category mappings are completed in `api-adatrack`.

### BigQuery field mapping

The BigQuery table `inventories-simplify-prod` uses singular field names (`category`, `subcategory`) that map 1:1 to `InventorySearchResult`. The table contains ~37 fields organized as: identifiers (`id`, `sku_equip`), product info (`product`, `description`, `brand`, `unity`, `stock`), classification (`category`, `subcategory`, `cluster`, `compilation`), 13 attribute fields (`color`, `presentation`, `type`, `model`, `size`, `measure`, `thickness`, `weight`, `volume`, `angle`, `fabrication`, `material`, `reference`), pricing (`price`, `price_b2b_def`, `price_b2b_inf`, `price_b2c_def`, `price_b2c_inf`, `currency`), media (`image0`–`image3`, `techsheet_url`), search (`keywords` as array), and metadata (`account_id`, `synced_at`).

The `keywords` field contains Peruvian construction synonyms (e.g., "fierro" → "barra de acero") generated by `api-adatrack` during sync. See `docs/PLAN_KEYWORDS_SYNONYMS.md` for the full synonyms dictionary.

### Endpoints

| Endpoint | Purpose |
|----------|---------|
| `GET /api/v1/search` | Search inventories via Vertex AI Search |
| `GET /api/v1/search/summary` | Search with AI-generated summary |
| `GET /api/v1/inventories/schema` | Get required_fields + field_options for a query |
| `GET /api/v1/schemas/catalog` | Full catalog of categories/subcategories with field details |
| `POST /api/v1/schemas/reload` | Force reload schemas from GCS |
| `GET /api/v1/schemas/status` | Cache status info |

### Cloud Storage schema

The inventory schema JSON lives at `gs://{GCS_BUCKET_KNOWLEDGE}/{GCS_INVENTORY_SCHEMAS_PATH}`. It contains `subcategory_schemas` with `required_fields`, `field_options`, and `category` per subcategory. If `category_schemas` is not present in the JSON, the service builds it automatically by grouping subcategories by category. Generated by a Colab notebook analyzing the BigQuery `inventories-simplify-prod` table. Current schemas cover 3 subcategories: Cables (Electricidad), Barras de Acero and Clavos (Acero).

### Vertex AI Search setup

Both a **Datastore** (stores/indexes data from BigQuery) and an **App/Engine** (activates search, generates `servingConfig`) must exist in Vertex AI Search. The code only references the datastore ID (`VERTEX_SEARCH_DATASTORE_ID`) — it accesses the serving config via the datastore path directly (`dataStores/{id}/servingConfigs/default_search`). The `VERTEX_SEARCH_COLLECTION` is always `default_collection`.

**Datastore schema configuration** — Fields in the datastore schema must be explicitly marked as:
- **Searchable** — used by the search engine for ranking (enable for: `product`, `description`, `brand`, `category`, `subcategory`, `keywords`)
- **Retrievable** — returned in API responses via `struct_data` (enable for all fields needed in the response)
- **Indexable** — allows exact-match filtering (enable as needed)

Enterprise edition and Generative Responses must be enabled on the App for semantic search to work.

## Environment Variables

Defined in `src/core/config.py`. Required:

- `GOOGLE_APPLICATION_CREDENTIALS` — path to GCP service account JSON (not needed on Cloud Run)
- `VERTEX_SEARCH_LOCATION`, `VERTEX_SEARCH_DATASTORE_ID`, `VERTEX_SEARCH_COLLECTION` — Vertex AI Search config
- `GCS_BUCKET_KNOWLEDGE` — Cloud Storage bucket name

Hardcoded in config: `GCP_PROJECT_ID`, `GCS_INVENTORY_SCHEMAS_PATH`, `GCS_TTL_SECONDS`.

## Deployment

Runs on **Google Cloud Run**. Cloud Run is detected via the `K_SERVICE` env var (see `src/core/environment.py`). Production uses gunicorn.
